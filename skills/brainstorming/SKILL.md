# Brainstorming — 把想法变成设计

## 概述

在写任何代码之前，先把设计想清楚。通过自然对话理解需求，形成方案，编写 spec 文档。

<HARD-GATE>
在用户批准设计方案之前，不得写任何代码、创建任何项目文件、或进行任何实现工作。这条规则对所有项目适用，无论看似多简单。
</HARD-GATE>

## Checklist

1. **探索项目上下文** — 查看文件、文档、最近的修改
2. **提问澄清** — 一次一个问题，理解目的、约束、成功标准
3. **提出 2-3 种方案** — 给出推荐和理由
4. **呈现设计** — 逐节呈现，每节获得用户确认
5. **编写 spec 文档** — 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
6. **Spec 自审** — 检查占位符、内部矛盾、范围漂移、歧义
7. **用户审查 spec** — 请用户 review 文档后确认
8. **进入下一阶段** — 进入 writing-plans

## 流程

### 理解需求

- 先看项目上下文（不要假设你了解项目）
- 评估范围：如果请求包含多个独立子系统，先分解
- 一次只问一个问题
- 优先用选择题，但也接受开放回答
- 聚焦：目的 → 约束 → 成功标准

### 探索方案

- 提出 2-3 种不同方案
- 给出推荐方案和理由
- 讨论权衡

### 呈现设计

- 从你认为理解清楚的地方开始呈现
- 每个章节的篇幅根据复杂度决定：简单就两三句话，复杂就 200-300 字
- 每讲完一节就问"这个方向对吗？"
- 覆盖：架构、组件、数据流、错误处理

### 设计原则

- **每个模块一个职责** — 能回答：它做什么、怎么用、依赖什么
- **YAGNI 狠一点** — 从设计中去掉所有不必要的功能
- **小文件优先** — 如果一个文件变大了，往往意味着它做了太多事
- **遵循现有模式** — 在已有项目中工作时，跟随既有的结构

### 编写 spec

- 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- 规格必须包含：概述、架构、组件设计、约束、错误处理

### Spec 自审

完成后自审：

1. **占位符扫描** — 有 "TBD"、"TODO" 或未完成的章节？修掉。
2. **内部一致性** — 章节之间有没有矛盾？架构描述和功能描述一致吗？
3. **范围检查** — 这个 spec 适合一个实施计划吗？还是要再分解？
4. **歧义检查** — 有没有可以有两种理解的需求？如果有，选一种并明说。

### 用户审查

> "Spec 写好了，保存在 `<path>`。请 review，需要修改的地方告诉我。确认后我们进入写实施计划。"

等用户反馈。用户要求修改就修改，重新自审。用户批准后进入下一阶段。

## 关键原则

- 一次一个问题
- 多选优先
- YAGNI 狠一点
- 每个模块单一职责
- 逐步确认，不要一口气讲完
- 灵活调整

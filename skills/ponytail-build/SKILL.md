# Ponytail Build — 懒人实现

## 概述

按实施计划逐个完成任务。每个任务内部采用测试驱动、最小实现、根因检查、子代理审查的完整闭环。

**核心理念：** 最好的代码是从未写过的代码。但在你必须写的时候，用最懒（高效）的方式写。

## 前提

- 必须有已批准的 plan 文档
- 必须在 Git 仓库中工作

## 每个任务的内部流程

```
Task N: [组件名]
│
├─ [ ] Step 1: 理解任务
│    ├─ 阅读 plan 中该任务的描述和接口定义
│    └─ 遵循 ponytail 规则：先检查是否已有实现
│
├─ [ ] Step 2: 写测试
│    └─ 按 plan 中的测试用例写测试文件
│
├─ [ ] Step 3: 运行测试（预期失败）
│    └─ 确认测试因未实现而失败
│
├─ [ ] Step 4: 写最小实现
│    └─ 遵循 ponytail 7 条阶梯（见后）
│
├─ [ ] Step 5: 运行测试（预期通过）
│
├─ [ ] Step 6: 根因检查
│    └─ 如果测试失败，必须先找根因再修复
│    └─ 修完后检查是否有其他调用者受影响
│
├─ [ ] Step 7: 子代理审查
│    └─ 创建一个新代理审查代码
│    └─ 通过 → 下一步 / 有问题 → 回 Step 4
│
├─ [ ] Step 8: 验证完成
│    └─ 最终运行所有测试
│    └─ 确认无遗留问题
│
└─ [ ] Step 9: 提交
     └─ git commit
```

## Ponytail 7 条阶梯

写任何代码之前，先走一遍：

1. **真的需要写吗？** （YAGNI — 也许根本不需要这个功能）
2. **代码库里已有？** （复用已有的工具函数、模式）
3. **标准库能搞定？** （Python 标准库很强大）
4. **平台原生特性覆盖了？** （操作系统、Claude Code 原生功能）
5. **已安装的依赖能解决？** （别重复造轮子）
6. **能一行搞定？** （列表推导、标准库一行函数）
7. **好吧，写最少的代码。** （最少、最直接、最无聊的实现）

## 子代理审查机制

### 触发

每个任务完成、测试通过后、提交之前。

### 审查指令

创建一个新的独立子代理，给它以下上下文：

```markdown
你是代码审查员。请审查以下代码变更。

【原始需求】spec.md 中对该功能的描述
【实施计划】plan.md 中该任务的步骤
【代码变更】修改了以下文件：
  - [文件路径] — 新增/修改内容

请检查：
1. 逻辑错误：代码是否能正确实现 spec 中的需求？
2. 边界情况：输入验证、错误处理是否充分？
3. 安全隐患：是否有注入、数据泄露等风险？
4. 完整性：spec 中所有功能点是否都已实现？
5. 过度工程：是否有不必要的抽象或代码？

不关注：
- 代码风格（除非明显不一致）
- 性能（除非有重大问题）
- 注释和文档（除非缺失关键说明）

请输出审查结论：通过 / 需修改（列出具体问题）
```

### 审查结果

- ✅ **通过** → 进入 Step 8 验证完成
- 🔄 **需修改** → 回到 Step 4，修复后重新提交审查

### 审查失败处理

如果子代理审查连续 3 次不通过：
1. 停止循环
2. 向用户报告情况
3. 请用户介入判断

## 验证完成标准

在宣布"完成"之前，必须确认：
1. ✅ 所有测试通过并有输出证据
2. ✅ 子代理审查通过
3. ✅ 无已知的边界情况未处理
4. ✅ 所有改动有对应的测试覆盖

## 平台适配

如果当前平台不支持创建子代理（无 Agent 工具可用）：
- 改为自我审查流程：重新阅读自己的代码，寻找问题
- 在审查结果中标注"自我审查（平台限制）"
